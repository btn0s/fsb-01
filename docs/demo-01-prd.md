## Project: Roy FSB Demo – Memory + Orchestration

### 1. Goal

Prove that Roy can:  
**Open‑ended text → use company memory (hybrid search) → advise on direction → generate UI variants → open draft PR with preview.**  
Ignore voice; assume text input. Next.js app on Vercel.

---

### 2. Core User Story

FSB/tech lead types:

> “We’re redoing onboarding for new full‑stack builders. I’m not sure whether heavy questionnaire or starter project is better. What fits our goals? Can you mock options I can share?”

Roy should:

1. Retrieve relevant OKRs / team goals / past decisions.
2. Propose 2–3 options, each tied to retrieved context.
3. When user picks an option, turn it into a structured `ChangeSpec`.
4. Use `ChangeSpec` to generate UI variants (v0).
5. Create draft PR in existing repo; Vercel auto‑creates Preview Deployment.
6. Surface spec, variants, PR + Preview links in Roy.

---

### 3. Minimal Memory & Hybrid Search

**Data (seeded manually for demo):**

- `docs/okrs/company-q3.json` (company OKRs).
- `docs/goals/fsb-team.json` (team goals).
- `docs/decisions/onboarding-experiment.md` (past decision: multi‑step onboarding failed).
- `docs/experiments/activation-funnel.md` (experiment summary).

**Storage:**

- Simple Postgres table (e.g., Vercel Postgres or Supabase):[1]

```sql
documents(id, title, type, tags, content, text_vector, metadata_json)
```

- `text_vector` is an embedding column (pgvector).
- Hybrid search = **BM25 (full‑text) + vector**:
  - Use `to_tsvector` for lexical ranking.
  - Use `cosine_distance` on `text_vector`.
  - Combine via weighted score in SQL or app code.[2][3]

**Indexing flow (one‑time script):**

- For each doc:
  - Compute embedding with LLM embedding API.
  - Insert row with `content`, `tags`, `metadata_json`, `text_vector`.[1][2]

### 4. APIs (Next.js App Router)

All routes implemented as server actions or API routes.

#### 4.1 `POST /api/advise` (question → context + options)

Input:

```ts
{
  question: string;
}
```

Steps:

1. **Hybrid search:**

   - Use `question` as query.
   - Run SQL that:
     - Filters by tags if any guessed from question (e.g., `onboarding`, `activation`).
     - Computes BM25 score + embedding similarity.
     - Returns top ~5 docs (title, type, snippet, full content).[3][2]

2. **LLM call:**
   - System: “You are Roy, a staff PM/tech lead with access to company memory.”
   - User: question + retrieved docs (titles + short snippets + types).
   - Ask for JSON:

```ts
type AdviseResponse = {
  contextUsed: {
    id: string;
    title: string;
    type: string;
    reason: string; // why relevant
  }[];
  options: {
    id: string; // "A" | "B" | "C"
    label: string; // short name
    description: string;
    rationale: string;
    risks: string[];
    recommended: boolean;
  }[];
};
```

Output: `AdviseResponse`.

#### 4.2 `POST /api/change-spec` (question + chosen option + context → ChangeSpec)

Input:

```ts
{
  question: string;
  chosenOption: AdviseResponse["options"][number];
  contextUsed: AdviseResponse["contextUsed"];
}
```

Prompt LLM to output this exact type:

```ts
type ChangeSpec = {
  title: string;
  summary: string;
  user_story: string;
  acceptance_criteria: string[];
  surfaces: ("web" | "mobile")[];
  target_route?: string;
  target_components?: string[];
  constraints?: string[];
  metrics_focus?: string[];
  priority?: "now" | "next" | "later";
};
```

Return `{ changeSpec: ChangeSpec }`.

#### 4.3 `POST /api/generate-prototypes` (ChangeSpec → v0 variants)

Input:

```ts
{
  changeSpec: ChangeSpec;
}
```

Steps:

- Compose prompt for **v0 Platform API**:[4][5][6]
  - Repo context (static string: Next.js app, Tailwind, shadcn, etc.).
  - ChangeSpec JSON.
  - Ask for 2–3 UI variants for the specified route/component.
  - Ask v0 to respond with file paths + contents, optionally demo URLs.

Output:

```ts
{
  variants: {
    id: string;
    name: string;
    files: { path: string; content: string }[];
    demoUrl?: string;
  }[];
}
```

#### 4.4 `POST /api/create-draft-pr` (variant → GitHub PR)

Input:

```ts
{
  variantId: string;
  files: {
    path: string;
    content: string;
  }
  [];
  changeSpec: ChangeSpec;
  contextUsed: AdviseResponse["contextUsed"];
}
```

Steps (GitHub REST API):[7]

- Create branch from `main`.
- Write files (`PUT /repos/:owner/:repo/contents/:path`).
- Create draft PR: `POST /repos/:owner/:repo/pulls` with:
  - Title: `ChangeSpec.title`.
  - Body:
    - Summary of ChangeSpec.
    - List of `contextUsed` (doc titles + reasons).
    - Note that UI generated by Roy via v0.

Output:

```ts
{
  prUrl: string;
  branchName: string;
}
```

Vercel’s GitHub integration auto‑creates a Preview Deployment for the PR.[8][9][10]

(Optional) `GET /api/preview-url?pr=number` to fetch preview URL from PR checks.[11][12]

### 5. Frontend UX (Next.js + shadcn)

Single page, three main sections:

1. **Question input**

   - Textarea “What are you trying to decide or build?”
   - Button: “Ask Roy”.
   - On click → `POST /api/advise`.

2. **Advice & memory**

   - Show:
     - “Here’s what Roy pulled from company memory” (list of `contextUsed` titles + types).
     - Cards for `options` (A/B/C) with label, rationale, risks, highlight `recommended`.
   - Each card has “Choose this option”.
   - On choose:
     - Call `POST /api/change-spec`.
     - Then auto‑call `POST /api/generate-prototypes`.

3. **Build outputs**

   - **ChangeSpec panel:**
     - Render title, summary, user story, AC list, surfaces, metrics.
   - **Prototype variants:**

     - Tabs/cards per `variant`.
     - Show demo iframe if `demoUrl` from v0, or code snippet preview.[4]
     - Each has button “Create draft PR from this variant”.

   - On “Create draft PR”:
     - Call `/api/create-draft-pr`.
     - When done, show:
       - “Open Draft PR” (GitHub).
       - “Open Preview” (optional preview URL when implemented).
       - “Open in Cursor” (deeplink string using PR number + repo).[13][14][15]

---

### 6. Non‑Goals

- No auth/multi‑tenant. Single demo repo and memory corpus.
- No streaming, STT, or TTS; all input is plain text.
- No long‑term persistence beyond what’s needed for the demo (docs table + PRs in GitHub).

---

### 7. Success Criteria

- Given a realistic onboarding question, `/api/advise` surfaces the right OKRs/decisions and 2–3 sensible options tied to them.[2][3]
- One click from option → ChangeSpec JSON that feels like a real PM/eng artifact.
- v0 returns at least 2 valid TSX variants compatible with the demo repo.[5][4]
- Draft PR is created with those files, and Vercel Preview loads successfully.[9][10][8]
- Entire flow runs live in < ~1 minute end‑to‑end on stage.
